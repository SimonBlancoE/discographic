Executive Summary

The recent visual work is generally coherent, but it introduced a new layer of duplication around stats fetching, stats-shape interpretation, feature metadata, and visual tokens. The most important problem is not the CSS or the carousel itself. It is that the app now fetches the same stats payload in more than one place and repeatedly re-derives small slices of the same data for the badge, hero, dashboard cards, and achievements. That duplication will make future UI changes more expensive and easier to desynchronize.

The best consolidation target is a small stats domain layer:
- one fetch/caching entrypoint
- one set of selectors for top genre, top artist, coverage counts, and badge genres
- one shared collection link/query builder for dashboard drilldowns

Everything else is secondary to that.

1. Concrete Duplication Findings

1.1 Duplicate stats fetches for the same session state

Severity: High

Evidence:
- src/App.jsx:20-43
- src/pages/Dashboard.jsx:195-224

Details:
- AppLayout fetches /stats only to derive badge genres for the header.
- Dashboard fetches /stats again for the main page.
- Both flows gate on the same account state and both degrade to an empty/absent result when Discogs is unavailable.

Why this matters:
- This is duplicated network work on the critical path of the logged-in shell.
- It duplicates account-state branching and response-shape assumptions.
- It makes the header badge and dashboard vulnerable to drifting behavior if one side starts filtering or normalizing the stats response differently.

Current duplicate logic:
- src/App.jsx:23-38 decides when to fetch and slices payload.genres.
- src/pages/Dashboard.jsx:196-218 decides when to fetch and stores the full payload.

What should exist instead:
- A shared stats hook or provider, for example:
  - src/hooks/useStats.js
  - src/lib/statsClient.js
  - src/lib/StatsContext.jsx
- If the app wants to keep the header light, a dedicated summary endpoint or selector layer could also work:
  - api.getStatsSummary()
  - selectBadgeGenres(stats)


1.2 Repeated interpretation of the stats payload shape

Severity: High

Evidence:
- src/App.jsx:28-32
- src/pages/Dashboard.jsx:71-79
- src/pages/Dashboard.jsx:144-148
- src/pages/Dashboard.jsx:246-279
- src/lib/achievements.js:167-179
- src/components/VinylBadge.jsx:14-37

Details:
- The same response structure is interpreted in several places:
  - top genres for the badge
  - top genre count/name for dashboard stat cards
  - total/rated/notes/priced coverage for readiness
  - top artist/top genre counts for achievements
  - genre entry normalization in VinylBadge

Symptoms:
- App assumes stats.genres can be sliced directly.
- Dashboard assumes stats.totals, stats.styles, stats.artists[0], and stats.genres[0].
- buildAchievements pulls counts and list lengths again from the raw stats object.
- VinylBadge accepts multiple input shapes and normalizes them locally, which means the presentational component is compensating for upstream contract looseness.

Why this matters:
- The payload contract is effectively "whoever remembers the field names."
- Any backend stats shape change will fan out across App, Dashboard, achievements, and the badge.
- It makes tests more fragmented because behavior is derived in many places instead of one.

What should exist instead:
- A shared stats adapter/selectors module, for example:
  - src/lib/stats.js
  - selectBadgeGenres(stats)
  - selectTopArtist(stats)
  - selectTopGenre(stats)
  - selectCoverageTotals(stats)
  - selectMappedStyleCount(stats)
- VinylBadge should ideally receive already-shaped display data or a narrower contract rather than normalizing arbitrary values itself.


1.3 Dashboard drilldown URLs are built ad hoc despite shared filter primitives already existing

Severity: Medium

Evidence:
- src/pages/Dashboard.jsx:187-193
- src/pages/Dashboard.jsx:391-395
- src/lib/api.js:56
- shared/collectionFilters.js:1-44

Details:
- Dashboard builds collection URLs manually with new URLSearchParams in two places.
- The collection feature already has shared filter keys/defaults in shared/collectionFilters.js.
- Collection itself already centralizes active filters through getActiveCollectionFilters.

Why this matters:
- The app already started consolidating collection-filter semantics, but dashboard drilldowns stopped short of using that shared layer.
- If filter names, normalization rules, or empty-value rules change, dashboard links can drift from collection state handling.

What should exist instead:
- Extend shared/collectionFilters.js with one link/query helper, for example:
  - buildCollectionSearch(filters)
  - createCollectionSearchParams(filters)
- Use that helper from Dashboard, Collection, and any future chart drilldowns.


1.4 Dashboard feature metadata has no shared source of truth

Severity: Medium

Evidence:
- src/components/HeroCarousel.jsx:5-6
- src/components/HeroCarousel.jsx:41-46
- shared/i18n.js:198-221
- shared/i18n.js:657-680
- tests/i18n-columns.test.js:68-94

Details:
- HeroCarousel owns FEATURE_KEYS = ['charts', 'random', 'curate', 'wall', 'export'].
- shared/i18n.js manually defines four keys per feature per locale:
  - kicker
  - title
  - sub
  - where
- tests/i18n-columns.test.js manually repeats the same matrix as VISUAL_I18N_KEYS.

Why this matters:
- Adding a new slide or renaming one currently means editing:
  - HeroCarousel feature list
  - Spanish message keys
  - English message keys
  - the test manifest
- That is not catastrophic, but it is exactly the kind of lightweight duplication that causes maintenance misses.

What should exist instead:
- A tiny shared manifest, for example:
  - shared/dashboardFeatures.js with ['charts', 'random', 'curate', 'wall', 'export']
- Tests should derive the expected translation keys from that manifest.
- The locale content still needs separate Spanish/English values; only the feature identity list should be shared.


1.5 Achievements maintain a second localization/content system outside shared/i18n.js

Severity: Medium

Evidence:
- src/pages/Dashboard.jsx:282
- src/lib/achievements.js:5-118

Details:
- Dashboard uses the main i18n system for the page shell.
- buildAchievements uses ACHIEVEMENT_COPY, a separate bilingual copy registry embedded in src/lib/achievements.js.

Why this matters:
- This is duplicated localization infrastructure, not just duplicated strings.
- It bypasses the existing translation coverage checks in shared/i18n.js.
- It creates two places where locale-aware copy is stored and maintained.

Why it is relevant to the latest work:
- The updated dashboard now leans harder on achievements as a primary section.
- That increases the cost of having achievement copy live outside the shared content system.

What should exist instead:
- Either:
  - move achievement strings into shared/i18n.js
- Or:
  - create a dedicated shared content registry used consistently by locale and tests

The current split is workable, but it is not clean.


1.6 Visual tokens are repeated as literals across CSS and component code

Severity: Medium

Evidence:
- src/index.css:124-175
- src/index.css:188-290
- src/index.css:293-467
- src/components/VinylBadge.jsx:5-11

Details:
- The new visual layer repeats the same brand colors and gradient language in several places:
  - hero panel surfaces
  - hero orbs
  - badge shell
  - carousel active dots
  - badge swatches

This is not all "bad duplication." Some variation is intentional. The issue is that the system now has enough custom visual language that it would benefit from one token layer for:
- pink/cyan brand ramps
- motion durations/easing
- reusable gradients for accent pills/dots/badge shells

Why this matters:
- Theme changes now require editing both CSS and JS constants.
- The badge swatch palette lives in component code while the rest of the visual identity lives in CSS.

What should exist instead:
- Prefer CSS custom properties in src/index.css for shared color ramps/durations.
- If the badge swatches must stay data-driven in JS, at least move them to a shared visual-tokens module.

4. Prioritized Recommendations

Priority 1
- Introduce a shared stats layer and stop fetching /stats independently in App and Dashboard.
- Move payload-shape interpretation into selectors.

Why first:
- Highest maintenance cost.
- Prevents future drift between badge, dashboard, and achievements.
- Removes the biggest duplication introduced by the recent visual work.


Priority 2
- Add a shared collection query builder and use it for dashboard drilldowns.

Why second:
- Small change, immediate cleanliness win.
- The shared filter layer already exists, so this finishes that consolidation.


Priority 3
- Extract a shared dashboard feature manifest and derive visual i18n coverage checks from it.

Why third:
- Small, low-risk consolidation.
- Directly reduces multi-file bookkeeping for the carousel.


Priority 4
- Consolidate achievement copy into the same localization system or a validated shared registry.

Why fourth:
- More structural than urgent.
- Important because achievements are now a prominent dashboard surface.


Priority 5
- Introduce visual tokens only where repetition is stable and intentional.

Why fifth:
- Worth doing, but only after the data/domain duplications are fixed.
- This is design-system cleanup, not the main maintainability bottleneck.


Priority 6
- Leave HeroCarousel/VinylBadge motion logic separate for now.

Why:
- There is some superficial repetition, but consolidation would currently trade clarity for abstraction.


Bottom Line

The main DRY issue in the current worktree is not "too many CSS classes" or "too many JSX nodes." It is repeated ownership of the same stats data and repeated interpretation of the same stats contract. Fix that first. After that, the next best consolidations are the collection query builder and the dashboard feature manifest. Everything else should be treated as selective cleanup, not blanket abstraction.
