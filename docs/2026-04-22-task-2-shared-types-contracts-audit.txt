Method
- Read the new UI files and traced the data they consume.
- Followed those payloads back to API routes and shared helpers.
- Compared repeated prop, route, and derived-state shapes across the repo.
- Focused on implicit JS object contracts rather than literal TS types, since the repo is plain JS.

Executive summary
- The most important missing shared contract is the dashboard stats payload. The new visual work expands the number of places consuming raw stats, but the repo still treats that payload as an untyped bag of nested fields.
- The second most important gap is release shape centralization. Collection, random release, wall, and release detail all depend on overlapping but not explicit release contracts.
- The third most important gap is account/auth and async job state. These are partially normalized at runtime, but not defined once and shared, so duplication persists across route, client API, context, and component layers.
- One good pattern already exists: collection filters are centralized in shared/collectionFilters.js. That file is the clearest example of how other contracts should be treated.

1. Dashboard stats contract is the highest-priority missing shared type

What exists now
- The stats route returns a mixed-shape payload from server/routes/stats.js:115.
- Dashboard consumes that payload directly in src/pages/Dashboard.jsx:204, src/pages/Dashboard.jsx:249, src/pages/Dashboard.jsx:257, src/pages/Dashboard.jsx:265, src/pages/Dashboard.jsx:273, src/pages/Dashboard.jsx:335, src/pages/Dashboard.jsx:364, and src/pages/Dashboard.jsx:391.
- Achievements consumes the same payload directly in src/lib/achievements.js:162 and then reads many nested properties in src/lib/achievements.js:169-179.
- App now calls the same full stats endpoint just to populate the header badge with genres in src/App.jsx:28-32.

Where the shape is implicit or inconsistent
- totals uses snake_case keys:
  - total_records
  - total_value
  - rated_records
  - notes_records
  - priced_records
  - server/routes/stats.js:116-122
- topValue is camelCase at the collection level but contains release fields with snake_case names like release_id and estimated_value:
  - server/routes/stats.js:129-132
- lastSync uses snake_case database column names:
  - started_at
  - finished_at
  - records_synced
  - status
  - server/routes/stats.js:107-113
- displayCurrency is camelCase:
  - server/routes/stats.js:135
- chart arrays are not all the same datum shape:
  - genres, formats, labels, styles, decades are { name, count }
  - server/routes/stats.js:25-27, 54-80
  - growth is { month, count }
  - server/routes/stats.js:82-88
  - artists is { artist, count }
  - server/routes/stats.js:98-105

Why this is risky
- The new visual refresh increases coupling to this payload. Dashboard and badge rendering now rely on several parts of it, but there is still no single definition of what a valid stats response is.
- The route mixes naming conventions. That makes drift more likely when new consumers are added.
- App is fetching the full dashboard payload just to use payload.genres.slice(0, 5) in src/App.jsx:28-32. That is a strong sign that the response contract is too broad for some consumers.
- buildAchievements assumes the same payload semantics as Dashboard but lives in a separate domain helper. A future change to stats could silently break achievements without touching dashboard code.

- Either split badge data into a smaller endpoint, or at minimum expose a reusable contract helper like getTopGenresFromStats(stats) so App is not ad hoc slicing raw payloads.

2. Badge and chart datum shapes are duplicated and too forgiving

What exists now
- HeroCarousel builds slides from translated keys into objects with kicker, title, sub, where in src/components/HeroCarousel.jsx:41-46.
- VinylBadge accepts genres and normalizes each entry via string-or-object heuristics in src/components/VinylBadge.jsx:14-24 and src/components/VinylBadge.jsx:26-37.
- GenreChart, StylesChart, FormatChart, LabelChart, and DecadeChart all consume raw arrays with specific assumptions:
  - GenreChart expects { name, count } in src/components/charts/GenreChart.jsx:14-17
  - StylesChart expects { name, count } in src/components/charts/StylesChart.jsx:11-16
  - GrowthChart expects { month, count } and then derives { month, total, added } in src/components/charts/GrowthChart.jsx:30-35
- FilterPanel expects options = { genres, styles, decades, formats, labels } in src/components/FilterPanel.jsx:6-12.

Where duplication shows up
- VinylBadge is effectively declaring its own input contract instead of consuming the dashboard genre datum contract:
  - it accepts strings
  - it accepts objects with .name
  - it ignores .count
  - src/components/VinylBadge.jsx:14-24
- Chart components each embed their own assumptions about datum keys instead of relying on a shared chart datum family.
- GrowthChart invents a second internal chart contract, cumulative = { month, total, added }, but that shape is not named or reused.
- FilterPanel relies on implicit option bucket names that must line up with server/services/releaseFilters.js:76-81 and CollectionWall state in src/pages/CollectionWall.jsx:21-25.

Why this is risky
- The badge currently succeeds because stats.genres happens to be compatible with the local string-or-object fallback logic. That is convenient, not safe.
- Overly flexible input contracts hide upstream mistakes. If badge input drifts from { name, count } to something else, buildBadgeGenres may keep working while silently discarding useful fields.
- Filter option payloads are shared across collection and wall surfaces but not defined once, which makes future additions easy to miss.

Concrete candidate shared contracts
- DashboardFeatureSlide
  - kicker
  - title
  - sub
  - where
- BadgeGenre
  - name
  - bg
- NamedCountDatum
  - name
  - count
- GrowthPoint
  - month
  - count
- GrowthSeriesPoint
  - month
  - total
  - added
- CollectionFilterOptions
  - genres: string[]
  - styles: string[]
  - decades: number[]
  - formats: string[]
  - labels: string[]

Recommendation
- Treat NamedCountDatum as a shared core contract and use it explicitly for genres, styles, labels, formats, and decades where possible.
- Tighten VinylBadge so it expects one stable input type instead of string-or-object unions.
- If HeroCarousel remains data-driven, move the generated feature object shape into a tiny helper so the contract is obvious and testable.

- The current worktree’s most pressing contract problem is the stats payload, because the new dashboard and badge now depend on it in multiple places without one authoritative definition.
